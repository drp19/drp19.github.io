Intro¶

A popular characteristics of locations, especially cities, is walkability. Walkability is a measurement of how close by common destinations, such as stores, attractions, and workplaces. This can have important ramifications for health, the economy, and the environment.

The National Walkability Index is a measure created by the EPA to measure this quantitatively. This is a composite score of the intersection density (how many roads), public transit proximity, and employment/household use. For each census block group, there is a score from 1 to 20, with 1 being the worst walkability.

In this analysis, we'll look for relationships between walkability and a couple of other statistics.

Data Processing¶

From this we see we have a lot of data - 117 columns and 220k data points at most. With this much data, we need to filter the data to make it more usable. Using this legend, we can remove a lot of columns. Columns like D4C, D4D, D5CRI, etc. are intermediate steps for the final calculations, so we can remove them. Finding a relationship between anything already measured in the walkability index won't be constructive, since we already know that relationship.

We will remove rows with a state identifier over 60 - these are US territories where we don't have complete data. We also remove rows where the population is 0 - this is probably bad data, since we assume that a census area would have people in it.

These steps reduce the number of bad matches we get between our datasets.

NOTE: Some of these steps can break if you run them twice, since we get rid of rows in clean up that previous steps need.

We now need to combine these tables together so that we can do analysis on them. We'll try to add all of our data onto the walkability index.

First, lets add the life expectancy table. First, we split the county as it contains both county and state name. Some rows don't have a census tract, which are summary rows for counties or states, and we remove them.

We then merge this with our county mapping table downloaded earlier, associating each county and state name with a FIPS identifier. Some rows don't get matched, which results from bad data, so we remove them.

We strip leading 0's from the county and state, and convert the tract number by formatting it with 2 decimal digits, then removing the decimal point. Now, we can join the two tables together.

We see that we only have 118 rows that can't be matched. To fix these, we would need to understand more about each individual area, and some other columns are missing, so its safe to assume that this isn't perfect data and we can ignore it. This essentially calculates an inner join rather than a left join, but its nice to know how many rows we couldn't match.
